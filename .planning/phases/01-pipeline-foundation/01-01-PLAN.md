<task type="auto">
  <name>Task 1: Create hardware auto-detection module</name>
  <files>nanoquant/hardware.py</files>
  <action>
Create `nanoquant/hardware.py` with:

1. A `HardwarePlan` dataclass with fields:
   - `gpu_free_bytes: int` — usable VRAM after CUDA overhead reserve
   - `cpu_free_bytes: int` — usable CPU RAM (90% of available)
   - `disk_free_bytes: int` — free disk at offload directory
   - `cuda_available: bool`
   - `use_gpu_for_blocks: bool` — True if GPU has >2 GB usable VRAM
   - `hessian_on_disk: bool` — False for Phase 1 (Hessian diags are tiny)
   - `model_load_strategy: str` — one of "cpu", "disk_offload", "auto_dispatch"
   - `device: str` — "cuda" or "cpu"

2. A `probe_hardware(offload_dir: str = ".")` function that:
   - Calls `accelerate.utils.get_max_memory()` to get GPU and CPU memory
   - Calls `psutil.disk_usage(offload_dir).free` for disk
   - Reserves 1 GB GPU headroom for activations
   - Reserves 10% CPU headroom for system
   - Sets `use_gpu_for_blocks = True` if GPU usable > 2 GB
   - Sets `model_load_strategy = "cpu"` (block-by-block manual .to(device) is correct for quantization per research)
   - Returns a `HardwarePlan` instance
   - Handles the case where no GPU is present (cuda_available=False, gpu_free_bytes=0)

3. A `print_hardware_summary(plan: HardwarePlan)` helper that prints a one-line summary like:
   `"Hardware: GPU=10.2GB usable, CPU=28.4GB usable, disk=145.3GB free, strategy=cpu"`

Use `from dataclasses import dataclass` for HardwarePlan. Import `accelerate.utils.get_max_memory`, `psutil`, `torch`.

Per user decision: no manual device map configuration required — this module auto-detects everything.
  </action>
  <verify>
Run `python -c "from nanoquant.hardware import probe_hardware, HardwarePlan; p = probe_hardware(); print(p)"` — should print a HardwarePlan with realistic values and not crash.
  </verify>
  <done>
`probe_hardware()` returns a populated HardwarePlan on both GPU and CPU-only machines. `print_hardware_summary()` outputs a readable one-liner.
  </done>

<task type="auto">
  <name>Task 2: Create safetensors checkpoint module</name>
  <files>nanoquant/checkpoint.py</files>
  <action>
Create `nanoquant/checkpoint.py` with:

1. `save_quantized_checkpoint(all_quantized, output_dir, model_name, rank)`:
   - `all_quantized` is `dict[str, dict[str, Tensor]]` mapping `layer_name -> {U_bin, V_bin, s1, s2, d_in, d_out}`
   - Flatten to `{layer_name}.{tensor_key}` format as safetensors keys
   - Call `.contiguous().cpu()` on each tensor before saving
   - Save to `output_dir/quantized_factors.safetensors` using `safetensors.torch.save_file`
   - Include metadata dict: `{"model": model_name, "rank": str(rank), "num_layers": str(len), "format": "nanoquant_v1"}`
   - Write `output_dir/quantized_manifest.json` with `{"model", "rank", "layers": [layer_names]}`
   - Use dotted notation for future MoE compatibility (e.g., `experts.gate_up_proj.expert_3.U_bin` — no square brackets)

2. `load_quantized_checkpoint(checkpoint_dir)`:
   - Read manifest JSON to get layer list
   - Use `safetensors.safe_open(path, framework="pt", device="cpu")` for lazy loading
   - Parse keys by splitting on last `.` to get `(layer_name, tensor_key)`
   - Return `dict[str, dict[str, Tensor]]` matching the save format

3. Import: `safetensors.torch.save_file`, `safetensors.safe_open`, `json`, `os`, `torch`

The naming convention `{layer_name}.{tensor_key}` uses the same `model.layers.N.self_attn.q_proj` prefix format already used in the existing `quantize.py` block loop.
  </action>
